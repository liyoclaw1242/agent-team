情況變體總覽

主流程的 8 種常見偏離，每個變體都有獨立的詳細頁面。

主流程 (happy path) 描述「全新 repo + 第一個商模 issue」這個理想路徑。實務上系統會遇到下列變體；每一個都有自己的觸發條件、流程分歧、與主流程的接點。

V1
RLM 已存在，新需求進場
第二個以後的商業 issue（最常見）
arch-shape rlm-modeler (有條件)
V2
既有代碼導入 (legacy bring-in)
已有業務代碼但無 project-rlm/
rlm-modeler 人工 gate
V3
arch-shape 諮詢 advisor
分解時遇到「現況是什麼」的問題
arch-shape advisor
V4
架構決策需要 ADR
遇到架構級選擇 (技術棧 / 整合策略 / 一致性)
arch-shape rlm-modeler 人工 gate
V5
executor 反推 (Mode C)
spec 矛盾 / 不可達 / 違反 RLM
executor arch-shape round-3 升級
V6
RLM 與現實不符
executor 實作中發現 RLM 失準
executor rlm-modeler
V7
小型 bug fix 直接派工
issue 範圍小，不影響領域模型
executor 繞過風險
V8
arch-judgment 兜底
dispatcher 無法分類 / 衝突 / 異常狀態
arch-judgment 1-5% 觸發

如何閱讀變體頁面每頁的固定結構

每個變體頁面都按以下章節組織，所以你可以快速找到關心的部分：

章節內容
前置狀態進入這個變體之前，repo / RLM / issues 的狀態長什麼樣
觸發條件workflow 用什麼 signal 判定要走這個分支
詳細流程stage-by-stage 的操作 + label 變化 + agent 動作
分支與失敗模式變體內部的二級分歧、卡死狀況、人工介入點
與主流程的接點處理完之後如何匯回主路徑
為何這樣設計背後的紀律與權衡 (與四角模型的關係)
相關變體常一起出現或會升級為的其他變體

變體不是 bug，是系統的設計表面。流程能識別並有規律地處理它們，才表示路由規則健全。

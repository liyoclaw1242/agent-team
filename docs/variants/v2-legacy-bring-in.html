V2 · 接管現有專案

既有代碼導入 (legacy bring-in)

repo 已有可運行的業務代碼，但 project-rlm/ 不存在。rlm-modeler 進入 reverse-bootstrap 模式，從代碼逆向推導模型。

觸發條件
代碼存在 + RLM 不存在 + 需求或維護 issue 進場
主要角色
rlm-modeler (reverse-bootstrap mode)
偏離主流程
取代 Stage 3 — 從代碼推 RLM 而非從 business intake 推

前置狀態

典型場景：團隊用 agent-team 接管一個既有專案。

  • repo 有可運行的應用：路由、ORM models、API endpoints、UI 頁面
  • 沒有 project-rlm/ 目錄
  • 可能有部分文件 (README、ADR、口頭知識) 但散落各處
  • 新進場的 issue 可能是商業需求，也可能是「先把這個專案 onboard 進 agent-team」

觸發條件

1. 任何 issue 進場 (含特殊 issue: <!-- intake-kind: onboarding -->)
2. project-rlm/ 不存在                      → 是
3. repo 內存在可辨識的業務代碼              → 是
   (有 src/, 有 package.json/pyproject, 不是空專案)
→ 規則命中：派 rlm-modeler (reverse-bootstrap mode)

注意這條規則必須排在「business intake → bootstrap」之前判斷，否則會錯把已有代碼的專案當空專案處理，覆寫掉現有結構。

詳細流程

onboarding 觸發 issue #1

三種典型觸發來源：

  • 商業需求 issue：使用者直接開新需求，但專案還沒接 agent-team
  • onboarding issue：明確的「請建模這個專案」<!-- intake-kind: onboarding -->
  • 手動 trigger：人工開 issue 並直接標 agent:rlm-modeler
workflow 偵測代碼存在 dispatcher
triggerissue + 無 RLM + 有代碼

dispatcher 不能套主流程的 bootstrap 規則 (那條只適用空專案)，改派 rlm-modeler 進 reverse-bootstrap。

labels: agent:rlm-modeler arch-state:reverse-bootstrap

rlm-modeler 掃描代碼 rlm-modeler

rlm-modeler 不直接寫 RLM，先做結構性 read：

讀取的線索
  • 路由 / API 定義 — Express routes, FastAPI routers, Next.js pages → 反映 user-facing capabilities
  • ORM models / migrations — entity 名稱、欄位、關聯 → 反映 domain entities
  • Job / queue handler — 反映非同步流程 / domain events
  • 測試的 user journey — 反映實際使用情境
  • README / docs / ADR — 反映歷史決策 (如有)
  • 環境變數 / secrets — 反映外部整合 (Stripe / Twilio / S3...)
推導 contexts rlm-modeler

根據代碼分組推導 bounded contexts。常見啟發式：

  • 同一個資料表群被同一組 service 操作 → 一個 context
  • 不同 prefix 的路由 (/api/orders/* vs /api/users/*) → 不同 context
  • 明顯獨立的子系統 (notification / billing) → 各自 context
產出 (推導，標註 inferred)
contexts/
  orders/        ← 從 src/orders/, models/order.py 推得
    description.md
    inferred: true   ← 標註這是逆向推導
  users/         ← 從 src/auth/, models/user.py 推得
  notifications/ ← 從 src/notifications/, jobs/email.ts 推得
推導 glossary rlm-modeler

從 model 名稱、API schema、UI 標題收集領域詞彙：

  • 程式碼用 Order、UI 標題用「訂單」、文件用「Sales Order」 → glossary 條目「訂單 (Order)」
  • 同義詞衝突 (customer vs client vs user) → 標 ambiguous，留待人工釐清
推導 stories rlm-modeler

從 API endpoints 和 UI flow 反推用戶故事：

POST /api/orders       → story: 顧客下單
GET  /api/orders/:id   → story: 查詢訂單狀態
POST /api/orders/:id/cancel → story: 取消訂單

每個 story 標註 inferred-from 指向源代碼路徑，方便 review 時對照。

寫入 + 驗證 rlm-modeler
執行 (透過 rlm CLI)
  • rlm define-context <ctx> --inferred 系列
  • rlm define-glossary ...
  • rlm define-story ... --inferred-from=src/orders/handlers.ts
  • rlm validate

每個 inferred 條目都帶來源指標，方便人工 review 時逐項對照。

開 PR + 標註審查重點 PR

PR body 強調這是逆向推導：

## ⚠️ Reverse-bootstrap PR

這份 RLM 是從現有代碼逆向推導，請逐項確認：

- [ ] contexts 的邊界是否符合團隊心智？
- [ ] glossary 中標 (ambiguous) 的詞彙怎麼釐清？
- [ ] inferred stories 是否遺漏了非顯性流程 (cron / 後台 / 客服)？
- [ ] 有沒有「程式碼裡有但不該有」的東西需要在 RLM 標 deprecated？

labels: arch-state:awaiting-bootstrap-merge

⚠️ 人工 gate (重度)　Reverse-bootstrap PR 比 V0 (空專案 bootstrap) 更需要仔細審查 — 機器只看代碼，但程式碼不等於意圖。團隊腦中的 domain 知識、被 deprecated 但還在 repo 的東西、客服流程、合規限制等都不會出現在代碼裡。

分支與失敗模式

狀況後果處理
代碼結構混亂 (god module / 沒有清晰邊界) rlm-modeler 切不出乾淨的 contexts 產出單一 monolithic context + 留 comment 標出問題區塊；後續用 V1 漸進切細
有死代碼 / 已 deprecated 但未刪 誤推導出不該存在的 stories 由人工 review 標 deprecated: true，不直接從 RLM 移除 (保留歷史)
有 monorepo 多服務 需要先決定建模範圍 onboarding issue 必須指定 --scope=path/to/service，rlm-modeler 不自行猜
review 時發現嚴重不對勁 整份 RLM 走偏 關閉 PR，重新 onboarding 並提供更明確的 hint (e.g. 「忽略 legacy/ 目錄」)

與主流程的接點

Reverse-bootstrap PR merged 後，project-rlm/ 第一次出現在 repo 裡。從這裡開始，後續所有新需求都走 V1 (RLM 已存在)。

原本觸發 V2 的 issue (#1) 處理方式：

  • 如果是商業需求：merged 後 workflow 把 #1 重新 dispatch，從 V1 流程繼續
  • 如果是onboarding issue：merged 後直接關閉 #1，等待真正的需求 issue 進場

為何這樣設計

「代碼是事實，意圖不是」　rlm-modeler 從代碼能可靠推導出「系統做了什麼」，但無法可靠推導出「為什麼這樣做」和「應該做什麼」。後者只能由人類在 review 時補上 — 所以 V2 的人工 gate 比 V0 更厚。

不接受「自動 reverse-bootstrap」(機器自動 merge) 的理由：

  • 程式碼裡的命名可能誤導 (User 在 UI 是「會員」，在後台是「員工」)
  • 已決議要砍掉但還沒砍的功能會被當成 first-class story
  • 外部整合的真正用途常常只活在工程師腦中

把這些問題留給機器處理 = 把錯誤模型寫死成系統的 SSOT，後續所有 arch-shape 分解都會從錯處延伸。

agent-team · 當前工作流 · v1

兩層架構

Intake & Design 跑在 sweet-home 的 Persistent terminal（手動 slash commands）； Delivery Pipeline 由 workflow.yaml 自動 dispatch。兩層分開運作，以 GitHub label 作交接點。

Layer 1 · Persistent Terminal（手動）

Intake & Design

使用者在 sweet-home Persistent tab 開互動 claude session， 以 slash commands 驅動。
不走 workflow.yaml dispatch。

hermes-intake hermes-design
/signal-to-spec  ·  /decompose-spec  ·  /design-approval
Layer 2 · workflow.yaml（自動）

Delivery Pipeline

workflow engine 每 30 秒 poll GitHub issues， 依 dispatch rules 自動 spawn claude -p。
完全無人值守，label 驅動。

worker whitebox-validator blackbox-validator arbiter
max_in_flight: 1  ·  interval_sec: 30
接口點 hermes-design 產出 kind:workpackage agent:worker status:approved → workflow.yaml 接手，自動觸發 worker
Layer 1 · Persistent Terminal · Intake & Design
1
hermes-intake — Business Signal → Draft Spec Persistent hermes-intake

使用者在 Persistent terminal 跑 /signal-to-spec（signal-to-spec skill）。

skill: signal-to-spec
  • 讀 issue body 業務信號 → business-model-probe / deployment-constraints-probe sub-skills
  • 產出 intake-complete → 寫 Spec body（subdomain + intake metadata）
  • 或產出 intake-question → enqueue-message 到 Discord
  • 或產出 intake-decline → 標 cancelled
結束狀態
issue 帶 kind:spec agent:hermes-design status:proposed
孿生觀測（Persistent tab）
  • 互動式 xterm.js 終端，輸入/輸出即時可見（非 one-shot，無 call chain）
  • Persistent sidebar：session 顯示 running / frozen 狀態
  • 無 dispatch_log 記錄（不走 workflow.yaml）
2
hermes-design — Spec → WorkPackages Persistent hermes-design

使用者在 Persistent terminal 跑 /decompose-spec（decompose-spec skill）。

skill: decompose-spec
  • 讀 RLM + Spec body，切出 N 個 atomic WorkPackages
  • 每個 WP 帶 subdomain / spec / AC / impact_scope / ADR refs / deps
  • 建立 WP issues（kind:workpackage）
  • enqueue-message → Discord 通知 Design Approval
Design Approval gate
使用者在 Discord 回 approve all 或 approve N， intake-confirmation skill 翻 labels → WP 變成 agent:worker status:approved
孿生觀測（Persistent tab）
  • 互動式 xterm，decompose-spec 執行過程即時可見
  • HomeView Issues：WP issues 出現（15 min 刷新）
  • Blocking graph：deps 關係即時顯示
  • 無 dispatch_log 記錄（不走 workflow.yaml）
⚡ 接口點 — Design Approval 完成後，WP 帶上 kind:workpackage agent:worker status:approved。 workflow.yaml 的下一個 poll tick（≤30s）自動命中 dispatch rule → 派 worker。

W
worker — TDD 實作 + PR worker

dispatch 觸發：kind:workpackage agent:worker status:approved

pre_spawn
transition_status: approved → in-progress（防止重複派發）
skill: tdd-loop + scenario skills（needs_worktree: true）
  • 偵測技術棧 → 派對應 TDD sub-skill（vitest / xunit / gotest / playwright）
  • scaffold skills：nextjs / cloudflare-worker / aspnet-minimal / go-http
  • budget: $5.0 · model: claude-sonnet-4-6
  • 產出 kind: delivered（pr_url + branch + fact_commits）
  • 或 kind: self-declined（spec 有問題，ADR-0016）
on_result: delivered
  • 留 worker_summary_comment（PR URL + fact commits + verify cmd）
  • remove agent:worker → add agent:validator
  • transition_status: in-progress → delivered
  • push_branch_and_pr（base: main，closes_issue: false）
孿生觀測（One-shot）
  • 即時串流：git / pnpm / go 每個 tool call + input/output 可展開
  • Thinking blocks：TDD red→green→refactor 推理過程可讀
  • Structured Output：out.pr_url + out.fact_commits 完整 JSON
  • Run header：issue #N + worker + 即時 cost
  • push_branch_and_pr 成功/失敗不記錄
Wb
whitebox-validator — PR diff 靜態審查 whitebox-validator

dispatch 觸發：kind:workpackage agent:validator status:delivered

skill: whitebox-validator · budget: $2.0
  • code-review sub-skill：logic / type / security / edge case 問題
  • stress-test sub-skill（高複雜度 WP）：DB query plan / 並發風險 / N+1
  • 產出 kind: approved 或 kind: rejected（含 findings + verdict）
on_result: approved
remove agent:validator → add agent:blackbox-validator，status 維持 delivered
on_result: rejected → arbiter
remove agent:validator → add agent:arbiter
孿生觀測
  • Call chain：讀哪些檔案、跑哪些檢查全部可見
  • Structured Output：findings + verdict + sub_skills 清單
Bb
blackbox-validator — Sandbox E2E / API 測試 blackbox-validator

dispatch 觸發：kind:workpackage agent:blackbox-validator status:delivered

skill: blackbox-validator · budget: $2.5
  • e2e-browser-test：Playwright 操作 sandbox app，截圖為憑
  • api-contract-test：curl / HTTP probe 驗 AC
  • Failure 分類：implementation-defect（bounce 回 worker）vs ac-ambiguity（退回 hermes-design）
on_result: approved
transition_status: delivered → validated · 留「PR ready for merge」comment
on_result: rejected-* → arbiter
remove agent:blackbox-validator → add agent:arbiter
孿生觀測
  • Structured Output：failure_classification 欄位決定 arbiter 路由方向
⚠️ Human Merge Gate — blackbox approved → status:validated。dispatch rule 返回 no_action（awaiting human merge）。PR 需人工 review + merge。Merge 後 issue close → done。
Ar
arbiter — Recovery 路由 arbiter

dispatch 觸發：agent:arbiter（任何 reject 或 degrade 後加上）

三種輸出 · budget: $1.0
  • retry — out.retry_stage 決定路由：
    • worker → add agent:worker，reset status: approved
    • white-box → add agent:validator，status 維持 delivered
    • black-box → add agent:blackbox-validator，status 維持 delivered
  • escalate — enqueue-message → Discord，add agent:human-help，status:blocked
  • cancel — transition_status: cancelled
Retry 計數器
set_body_marker 更新 retry-worker / retry-white-box / retry-black-box
孿生觀測
  • Dispatch Log：arbiter spawn → retry spawn 形成清楚 recovery 路徑
  • Structured Output：out.decision + out.reasoning + out.failed_stage 完整可見
  • 同一 issue 多次 dispatch 記錄保留，retry 次數一目了然
Dg
Degrade — 無結構化輸出時 on_no_structured_output

任何 spawn 結束但 JSON 無法解析 → 固定路由到 arbiter。

步驟
留 degrade_comment（role / end_reason / cost / last_assistant_text）→ 清除所有 agent:* labels → add agent:arbiter → set_body_marker: last-degrade-role / last-degrade-end-reason
孿生觀測
  • One-shot chat：stderr（紅色）通常顯示無輸出原因
  • Dispatch Log：下一個 tick 出現 arbiter spawn
  • degrade_comment post 成功/失敗不記錄
Un
Unblock Pass — Dep 解鎖 unblock_pass: enabled

每次 poll 前執行，掃描 <!-- deps: #N #M --> markers，deps 全 close → 解鎖。

on_unblock
remove status:blocked → add status:approved → 留 comment
孿生觀測
  • Blocking graph：deps close 後 issue 自動消失
  • Dispatch Log：解鎖後下一 tick 出現 worker spawn
  • unblock pass 本身執行細節不記錄

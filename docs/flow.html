Delivery Pipeline

以 GitHub label 作狀態機，workflow YAML 宣告 dispatch 規則 + side effects。 六個角色分工，每個 spawn 是一次性 claude -p，無長駐進程。

主流程 · 階段說明

1
Hermes Intake — Spec 初始化 hermes-intake

觸發條件：kind:spec agent:hermes-intake status:proposed

pre_spawn
transition_status: proposed → in-progress
skill: signal-to-spec
  • 讀 issue body 中的業務信號（business-model-probe / deployment-constraints-probe）
  • 產出 intake-complete → 寫 Spec body + subdomain marker
  • 或產出 intake-question → 轉給 Discord（enqueue-message）
  • 或產出 intake-decline → 標 cancelled
on_result: intake-complete
  • 留 comment、remove agent:hermes-intake、add agent:hermes-design
  • transition_status: in-progress → proposed
孿生觀測
  • Dispatch Log：spawn · kind:spec #N · rule index
  • One-shot sidebar：role = hermes-intake（紫色 badge）
  • Call chain：signal-to-spec 讀 issue + 各種 probe 執行 tool calls 可見
  • intake-question enqueue_exit 成功或失敗不可見（on_result step）
2
Hermes Design — Spec → WorkPackages hermes-design

觸發條件：kind:spec agent:hermes-design status:proposed

注意：hermes-design 在 dispatch rules 中刻意不自動觸發。使用者在 Persistent terminal 手動執行 /decompose-spec slash command 啟動，需要人工參與 DesignApproval gate。

skill: decompose-spec
  • 讀 RLM + Spec，切出 N 個 atomic WorkPackage
  • 每個 WP 含 subdomain / spec / AC / impact_scope / ADR refs / deps
  • 產出 kind: decomposed → 批量 create_issue（kind:workpackage）
  • enqueue-message 通知 Discord 待 Design Approval
孿生觀測
  • Call chain：decompose-spec 讀 RLM + 構建 WP spec 的全過程可見
  • Structured Output：out.workpackages 的完整 JSON（N 個 WP 定義）
  • HomeView Issues：WP issues #N–#M 出現（15 min 或手動刷新）
  • Blocking graph：WP 之間的 deps 關係即時顯示
  • create_issue steps 成功/失敗不記錄；enqueue_exit 不可見
⚠️ Design Approval Gate — hermes-design 產出的 WP 預設帶 status:proposed，dispatch rules 要求 status:approved + agent:worker 才會觸發 worker。人工在 Discord 回 approve all 或 approve N 後由 intake-confirmation skill 翻 labels。
3
Worker — TDD 實作 + PR worker

觸發條件：kind:workpackage agent:worker status:approved

pre_spawn
transition_status: approved → in-progress（防止重複派發）
skill: tdd-loop + scenario skills
  • 偵測技術棧（package.json / go.mod / *.csproj）→ 派對應 TDD sub-skill
  • vitest-test-first / xunit-test-first / gotest-table-driven / playwright-test-first
  • scaffold-nextjs / scaffold-cloudflare-worker / scaffold-aspnet-minimal / scaffold-go-http
  • needs_worktree: true — 每次在獨立 git worktree 工作
  • 產出 kind: delivered（含 PR URL + branch + fact commits）
on_result: delivered
  • 留 worker_summary_comment（PR URL + fact commits + verify cmd）
  • remove agent:worker，add agent:validator
  • transition_status: in-progress → delivered
  • push_branch_and_pr（base: main，closes_issue: false）
孿生觀測
  • 即時串流：每一個 git / pnpm / go / dotnet tool call 逐一顯示 input/output
  • Thinking blocks：TDD 推理過程可展開（red → green → refactor 決策）
  • Structured Output：out.pr_url + out.branch + out.fact_commits 完整 JSON
  • Run header：issue #N + worker badge + 即時 cost 累積
  • Dispatch Log 新增 whitebox-validator spawn（on_result label 翻轉後下一個 tick）
  • push_branch_and_pr step 成功/失敗不記錄
4a
WhiteBox Validator — PR diff 靜態審查 whitebox-validator

觸發條件：kind:workpackage agent:validator status:delivered

skill: whitebox-validator + sub-skills
  • code-review：讀 PR diff，找 logic / type / security / edge case 問題
  • stress-test：高複雜度 WP 才啟動，分析 DB query plan / 並發風險 / N+1
  • 產出 kind: approved 或 kind: rejected（含 findings + verdict）
on_result: approved
remove agent:validator，add agent:blackbox-validator，status 維持 delivered
on_result: rejected → arbiter
remove agent:validator，add agent:arbiter
孿生觀測
  • Call chain：whitebox 讀哪些檔案、跑什麼檢查工具，全部 tool calls 可見
  • Structured Output：findings 列表 + verdict + sub-skills run
  • verdict 決定下一個 dispatch：Dispatch Log 出現 blackbox 或 arbiter spawn
4b
BlackBox Validator — Sandbox E2E / API 測試 blackbox-validator

觸發條件：kind:workpackage agent:blackbox-validator status:delivered

skill: blackbox-validator + sub-skills
  • e2e-browser-test：Playwright 操作 sandbox app，截圖為憑
  • api-contract-test：curl / HTTP probe 驗 AC 的 request/response shapes
  • Failure 分類：implementation-defect（bounce 回 worker）vs ac-ambiguity（退回 hermes-design）
on_result: approved
remove agent:blackbox-validator，transition_status: delivered → validated，留「PR ready for merge」comment
on_result: rejected-* → arbiter
remove agent:blackbox-validator，add agent:arbiter
孿生觀測
  • Call chain：Playwright / curl tool calls + assertion 結果可見
  • Structured Output：failure_classification 欄位決定 arbiter 如何路由
  • 完整通過後 Dispatch Log 出現 no_action（status:validated 不再觸發）
⚠️ Human Merge Gate — blackbox approved 後 issue 標 status:validated，dispatch rule 返回 no_action（awaiting human merge）。PR 需要人工 review + merge。Merge 後 issue close，WP done。
R
Arbiter — Recovery 路由 arbiter

觸發條件：agent:arbiter（任何 reject 或 degrade 後加上）

輸出三選一
  • retry：根據 out.retry_stage（worker / white-box / black-box）重置 label + status，回到對應 dispatch rule
  • escalate：enqueue-message → Discord，add agent:human-help，status:blocked
  • cancel：transition_status: cancelled
retry 路由細節
  • retry worker → add agent:worker，reset status: approved（worker dispatch 要求）
  • retry white-box → add agent:validator，status 維持 delivered
  • retry black-box → add agent:blackbox-validator，status 維持 delivered
  • set_body_marker 更新 retry 計數器
孿生觀測
  • Dispatch Log：arbiter spawn + 之後的 retry spawn 形成清楚的 recovery 路徑
  • Structured Output：out.decision + out.reasoning + out.failed_stage 完整可見
  • 同一 issue 的多次 dispatch 記錄全部保留，可看出 retry 幾次
D
Degrade Fallback — 無結構化輸出時 on_no_structured_output

觸發條件：任何 spawn 結束但沒有可解析的 JSON structured output。

固定處理
  • 留 degrade_comment（role / end_reason / cost / last_assistant_text 截斷）
  • 清除所有 agent:* labels
  • add agent:arbiter
  • set_body_marker：last-degrade-role / last-degrade-end-reason
孿生觀測
  • Dispatch Log：run 結束後下一 tick 出現 arbiter spawn
  • One-shot chat：stderr lines（紅色）通常顯示為何沒輸出結構化 JSON
  • Result bar：completed 但 structured output JSON 缺少 kind 欄位
  • degrade_comment 是 on_result step，post 成功/失敗不記錄
U
Unblock Pass — Dep 解鎖 unblock_pass: enabled

每次 poll 前執行。掃描所有帶 <!-- deps: #N #M --> 的 issues，當所有 dep issues 都已 close 時解鎖。

on_unblock steps
留 comment「Dependencies satisfied」，remove status:blocked，add status:approved
孿生觀測
  • Blocking graph：deps 關係即時更新，blocked issue 在 deps close 後消失
  • Dispatch Log：unblock 後下一 tick 出現 worker spawn（status:approved 命中 dispatch rule）
  • unblock pass 本身的執行過程不記錄（無 dispatch_log 條目）
  • on_unblock comment post 成功/失敗不記錄

Dispatch Rules · 完整列表（first-match）

# When（labels / state） Then 備註
0 issue_state: closed no_action — closed Terminal check 先行
1 has_label: status:cancelled no_action — cancelled
2 has_label: status:done no_action — done
3 has_label: human-review no_action — awaiting human review
4 has_label: agent:human-help no_action — blocked on Discord
5 has_label: agent:arbiter spawn_fresh — arbiter Recovery 最高優先
6 kind:spec + agent:hermes-intake + status:proposed spawn_fresh — hermes-intake
— kind:spec + agent:hermes-design + status:proposed （無 rule）手動觸發 /decompose-spec 刻意不自動 dispatch
7 kind:workpackage + agent:worker + status:approved spawn_fresh — worker
8 kind:workpackage + agent:validator + status:delivered spawn_fresh — whitebox-validator
9 kind:workpackage + agent:blackbox-validator + status:delivered spawn_fresh — blackbox-validator
10 kind:workpackage + status:validated no_action — awaiting human merge
— （none matched） implicit no_action

Label 狀態機

Status Label 進入條件 離開條件
status:proposed issue 建立時；intake-complete on_result；hermes-design pre_spawn（in-progress 後回到 proposed） worker pre_spawn → in-progress；hermes-intake pre_spawn → in-progress
status:approved hermes-design decomposed on_result；unblock_pass on_unblock worker pre_spawn → in-progress；arbiter retry worker → approved（reset）
status:in-progress worker pre_spawn；hermes-* pre_spawn worker delivered on_result → delivered；degrade → （清除並由 arbiter 處理）
status:delivered worker delivered on_result blackbox approved → validated；arbiter retry worker → approved
status:validated blackbox approved on_result 人工 merge → 手動 rlm mark-delivered → done
status:blocked arbiter escalate；intake-question；design-clarification unblock_pass（deps closed）→ approved；人工介入
status:cancelled arbiter cancel；intake-decline；worker self-declined；design-decline Terminal — 不再 dispatch

孿生觀測覆蓋率

● 完整可見

  • Dispatch 決策記錄（dispatch_log 即時推送）
  • 每個 spawn 的 role / issue# 連結
  • Tool call 序列（thinking / tool_use / tool_result）
  • Thinking blocks（字元數 + 可展開）
  • Structured output 完整 JSON（kind + 全部欄位）
  • Spawn prompt（argv + 完整 prompt body）
  • Cost per run（即時累積）
  • Issue blocking graph（deps 關係）

○ 暗區 · 不可見

  • on_result steps 執行結果（create_issue / comment / run_command 成功失敗）
  • pre_spawn Abort / Reroute（現有 workflow 均為 Continue，無 abort）
  • push_branch_and_pr 執行結果
  • unblock_pass 執行細節
  • enqueue-message 到 Discord 的成功/失敗
  • GitHub label 是人工加的還是 workflow 加的（無法區分）
  • Arbiter retry 後，error quarantine（human-review）原因

Runtime 注意事項（v1 已知限制）

v1 Simplifications（workflow YAML 開頭有記）
  • max_in_flight: 1 — Worker 和 Validators 互相 block。同一時間只跑一個 agent。Phase 2 改為 per-role semaphore。
  • Dep 自動解析 — 目前僅靠 issue body 中的 <!-- deps: #N --> markers，unblock_pass 解鎖。for_each iter result-number 自動 wiring 是 Phase 2。
  • Post-condition trust — 信任 agent 的 JSON（worker 說 delivered + pr_url，就信它）。Validators 是真正的驗證機制，不做二次確認。
  • Arbiter reads prior output via gh comments — 每個 on_result handler 都把 structured JSON echo 成 comment（paper trail）。Arbiter 靠讀 comments 得知上一個 agent 的輸出。Sweet-home 直接注入 prior output 到 prompt header 是 Phase 2。

GitHub label = 狀態機  ·  dispatch_log = 決策稽核  ·  run_events = 可回放 call chain  ·  SQLite = 觀測記憶

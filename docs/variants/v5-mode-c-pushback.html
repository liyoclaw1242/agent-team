V5 · 反推迴圈

executor 反推 (Mode C pushback)

executor 認為 spec 無法執行時，不硬做，而是留 pushback comment 退回 arch-shape — 但有 round counter，3 round 卡關就升級人工。

觸發條件
executor 收到子 issue 後判定 spec 矛盾 / 不可達 / 違反 RLM
主要角色
executor · arch-shape · 視情況人工
偏離主流程
主流程 Stage 7 退回 Stage 5；最多 3 輪

前置狀態

已進入主流程 Stage 7 (executor 拾起子 issue 開始實作)。executor 在開工前或途中發現問題：

  • spec 內部矛盾：要求 「同步回應」+「等待外部 webhook」
  • acceptance criteria 不可達：要求「100ms 內完成」但需要的查詢本身就 200ms
  • deps 缺失或衝突：依賴的子任務尚未存在 / 被誤標為已完成
  • 違反 RLM 約束：spec 要求改某個 entity，但 RLM 標明該 entity immutable
  • spec 引用不存在的 RLM slug：context 或 story 找不到

觸發條件

executor 在 spawn 開始時做 sanity check。任何一條 fail 就觸發 pushback：

1. 子 issue 的 RLM slugs 是否都能解析？
2. acceptance_criteria 是否內部一致？
3. deps 列出的 PR 是否真的都已 merged？
4. spec 是否與 RLM context / story 對齊？
5. 我能想出至少一個能滿足 acceptance criteria 的實作？
   (這條是最後 — 試著實作，發現做不到才退回)

任一 fail → 進入 Mode C pushback

詳細流程

executor 識別問題 executor

spawn 開始後，executor 先讀 spec 跑 sanity check。

範例：spec 矛盾
子 issue #X spec：
"處理付款 webhook，必須在 50ms 內回應 Stripe，
 並在回應前完成發票寄送 (含 PDF 附件)"

executor 判定：
- PDF 生成 + email 發送 ≥ 1 秒
- 不可能在 50ms 內完成這些步驟
- 結論：spec 內部矛盾，無法執行
executor 留 pushback comment executor

executor 不是抱怨 — 而是留結構化的 pushback，幫 arch-shape 修正。

<!-- mode-c-pushback -->
problem: spec 內部矛盾
detail: |
  spec 要求 "50ms 內回應 + 完成發票寄送"。
  發票流程包含 PDF 生成 (~500ms) 和 SMTP 發送 (~1-3s)。
  這些操作不可能塞進 50ms 視窗。

suggestions:
  - 拆兩個子任務：(a) webhook 同步 ack (b) 發票非同步處理
  - 或：放寬時間限制至 5s，但需要評估 Stripe retry 行為
  - 或：發票寄送降級為 fire-and-forget queue 任務

referenced_rlm:
  - payments.webhook-receiver (story)
  - billing.invoice-issuance (story)

executor_status: declined

executor 結束 spawn，不開 PR，標 mode-c-rounds:1。

workflow 派 arch-shape 重審 dispatcher
trigger子 issue 標 mode-c-rounds:N + 有 mode-c-pushback comment

workflow 把這個子 issue 退回 arch-shape，附上 executor 的 pushback comment 內容。

labels: agent:arch-shape + mode-c-rounds:1 保留

arch-shape 修正 spec arch-shape

arch-shape 讀 pushback，三種可能動作：

動作選擇
  • 修正 spec：直接改子 issue body，採納 executor 的建議 (e.g. 拆成兩個 sub-issue)
  • 補充資訊：原本 spec 寫得不夠，補 acceptance criteria / RLM 引用
  • 升級 ADR (V4)：發現背後是設計選擇，需要人類決定 → 走 V4
  • 反駁 executor：認為 pushback 不成立，留 comment 解釋並重派

修正完，標移除 agent:arch-shape，加 status:ready，重派 executor。

第二輪 executor executor

新的 executor spawn 拿到修正後的 spec，重新 sanity check：

  • ✅ 通過 → 開始實作 → 主流程 Stage 7 繼續
  • ❌ 仍有問題 → 第二次 pushback → mode-c-rounds:2

Round counter 升級邏輯

1 Round 1 → 2

第一次 pushback 是常態 — spec 第一版常常不完整。arch-shape 修正後通常能通。

典型原因：spec 模糊、缺 acceptance criteria。

2 Round 2 → 3

第二次仍卡 = spec 修正方向錯了，或者問題其實是設計選擇。

典型原因：背後其實需要 ADR (V4)，arch-shape 應升級而不是繼續硬切。

3 Round 3 → human-review

第三次仍卡 = LLM 之間互相說不通了。系統強制升級 — 標 human-review，停止自動 dispatch。

典型原因：根本性的問題，不是 spec 措辭問題。

⚠ 為何停在 3

避免無限拉鋸 — LLM 反覆改 spec 不會收斂時繼續砸 token 是浪費。3 是經驗值，可調但不該調太高。

❌ 不該繞過的反模式　executor 看到難 spec 時 硬實作 是最壞的選擇 — 產出對不上 acceptance criteria 的 PR，下游 review 還要花更多時間 debug。「pushback 是負責任的行為」這個紀律要寫進 executor 的 prompt 並在 review 時強化。

分支與失敗模式

狀況後果處理
executor 過度敏感，動不動就 pushback 系統一直卡在 round 1-2 review pushback comments 找 pattern；可能是 spec template 缺欄位
arch-shape 反駁 executor 但執理不通 第二次 spawn 又會 pushback round counter 自動處理 — 兩輪後升級 V8 (arch-judgment)
pushback 內容空洞，只說「做不到」 arch-shape 無法修正 pushback comment 必須符合 schema (problem / detail / suggestions)，否則退回 executor 重寫
同一個 spec 被多個 executor 拒 spec 真的有問題 不是輪數的問題；arch-shape 應主動升級 ADR 或諮詢 advisor

與主流程的接點

Round 1-2 解決後，子 issue 重新進入 Stage 7 (executor 實作)；後續主流程不變。

Round 3 升級 human-review 後，整個系統暫停處理該 issue (但其他無關的子任務繼續並行)。人類介入後三種出路：

  • 人類直接改子 issue body 解決 → 重置 round counter → 重派 executor
  • 人類拆 issue 為更小範圍 → 原 issue 關閉，新 issue 進入主流程
  • 人類認定該需求暫時不做 → 關閉子 issue，arch-shape 收尾時會發現缺塊，可能升級 V4 ADR

為何這樣設計

「拒絕」是基礎能力　executor 必須有「不執行」的選項，否則它會在 spec 不可能達成時硬寫一個對不上的實作。pushback + round counter 把這個能力結構化 — executor 不是抱怨，是用既定 schema 反推。

round counter 的設計理念：

  • 有限重試：機率上 LLM 第一次失誤可以修，但連續 3 次說不通就是設計問題不是表達問題
  • 清晰升級路徑：每個 round 都是嘗試把 spec 修對，第 3 round 就轉責給人類
  • 避免無限循環：兩個 stateless LLM 來回改，沒有外力介入永遠不會收斂
  • 留下審查痕跡：每輪 pushback 都是 GitHub comment，事後可以審 spec 為什麼這麼難寫

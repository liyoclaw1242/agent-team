V7 · 快速通道

小型 bug fix / 直接派工

issue 範圍小、不影響領域模型，跳過 arch-shape 直接派 executor。是合法的「快速通道」，但路由規則必須嚴謹，誤判會繞過 RLM 校準。

觸發條件
issue 明確小範圍，無領域語意改動
主要角色
executor (僅一個 spawn)
偏離主流程
跳過 Stage 5 (arch-shape 分解)，直接進 Stage 7

前置狀態

project-rlm/ 已存在，業務代碼運作中。新進場的 issue 看起來不像「商業需求」，而是不影響領域模型的小型修改：

  • typo / log 訊息修正
  • 視覺微調 (顏色、間距、文案)
  • 缺漏的 null 檢查 / try-catch
  • 明顯錯誤的條件邏輯 (off-by-one, 反向比較)
  • 升級單一 dependency 版本 (修 CVE)
  • 補測試覆蓋現有功能

這些修改特徵：不引入新概念、不改變 entity 結構、不影響 user flow。

觸發條件 (路由判準)

workflow dispatcher 用一組保守判準辨識 V7。所有條件都要成立才走快速通道：

檢查必須是
issue body 是否有 intake-kind: business？否
issue body 是否有 intake-kind: bugfix 或 chore？是
是否引用 RLM context / story slug？否
是否提到「新增」「設計」「重構」？否
預期改動規模 (issue body 標 scope)？≤ 1 file 或 ≤ 50 行
是否影響 schema / migration？否

任一條 fail → 不走 V7，落到主流程或 V8 (arch-judgment)。

❌ 誤判風險　路由規則的預設行為應該是「走主流程」。V7 是優化，不是常規路徑。一個「typo 修正」的 issue 如果其實隱含 i18n 重構，繞過 arch-shape 就會錯失機會。有疑慮就走全流程。

詳細流程

issue 進場 (明確小範圍) issue #N

例：

title: 修正登入頁面誤字「請輸入帳棕」應為「請輸入帳號」
body:
  <!-- intake-kind: bugfix -->
  <!-- scope: 1-file -->

  位置：src/pages/login.tsx 第 42 行
  目前顯示「請輸入帳棕」應為「請輸入帳號」
workflow 路由判定 V7 dispatcher
trigger所有 V7 判準都成立

跳過 arch-shape，直接派 executor。

labels: agent:executor status:ready flow:fast-track

flow:fast-track label 用來事後審計 — 知道哪些 PR 走了快速通道。

executor 直接實作 executor

沒有 RLM-grounded spec，但 issue body 本身包含足夠資訊 (檔案路徑 / 行號 / 預期結果)。executor 仍跑 sanity check：

V7 額外檢查
  • 實作改動是否真的限在宣稱的範圍？(實際 diff vs scope label)
  • 有沒有「順手」想加的東西？(規則：不加)
  • 修改是否引入新概念 / 新術語？(若是，立刻 escalate 為 V5 pushback，要求走主流程)

這層自我約束很重要 — 「我來修 typo」不該變成「順便重構這整個 component」。

PR + review + merge PR

標準 PR 流程，但 review 重點不同：

  • 確認改動仍在宣稱的 scope
  • 沒有偷渡領域邏輯改動
  • 不需驗證對齊 RLM (沒有 spec → 沒有對齊對象)

merged → issue 關閉。沒有 cascade unlock (因為沒有子任務)。

分支與失敗模式

狀況後果處理
executor 發現 issue 比想像中複雜 (typo 其實牽涉 i18n) scope 超過 V7 判準 executor 在 PR 開前發 escalate-to-main-flow comment → workflow 改派 arch-shape
使用者把商業需求標成 bugfix 想走快車道 繞過 arch-shape 校準 executor 看 issue body 發現有領域語意 → 同上 escalate
實作中發現需要改 schema 或 migration 違反 V7 判準 立刻 escalate；schema 變動絕對不該走 V7
多個 V7 改同一個 file 但意圖衝突 合併衝突 / 邏輯衝突 workflow 不允許同 file 並行 V7；後到的等前者 merge
累積很多 V7 之後才發現底層需要重構 tech debt 定期由人類審計 flow:fast-track PR，找重構機會 (不是 V7 的責任)

與主流程的接點

V7 不接主流程 — 它是獨立完成的迷你迴圈。完成後沒有 cascade、沒有 parent issue 收尾、沒有 RLM 變更。

但 V7 的升級路徑很重要：任何時候 executor 發現 scope 超出 V7 判準，必須立刻 escalate，回到主流程或 V5 pushback。這是防止 V7 被濫用的關鍵閥門。

為何這樣設計 (與為何要小心)

不是所有改動都需要全流程　堅持每個 typo 都走 RLM 校準 + arch-shape 分解 + 子任務切分是過度結構化，會讓系統實際上不可用。V7 承認「有些事就是改一行就好」，但用嚴格判準把快車道限縮在最安全的場景。

權衡的兩面：

  • 效率收益：小修不用走完整流程，能快速回應 typo / 視覺問題 / 緊急 hotfix
  • 校準損失：跳過 arch-shape = 跳過 RLM 對齊檢查；長期累積會讓 RLM 與代碼漂移

因此設計上的紀律：

  • 路由規則保守：所有判準都要成立才走 V7；任一不成立 → 主流程
  • executor 自查：實作中發現超出 scope 必須 escalate
  • 事後審計：flow:fast-track label 讓快車道 PR 可批次審視
  • 定期回顧：V7 占比應該 ≤ 30%；超過代表路由太鬆或主流程太重

什麼時候不該用 V7

  • 任何引入新術語 / 新概念的修改
  • schema / migration 改動
  • 跨多個 file 的「同一改動」(可能該拆，或可能該走主流程)
  • UI 設計重構 (即使「只改 CSS」)
  • 性能優化 (背後常常牽涉演算法選擇 = ADR 級)
  • 安全性修正 (應該走 security review，不只 fix 一行)

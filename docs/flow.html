agent-team

觀測側 · 孿生

agent-sweet-home

Tauri 2 桌面 app。把 GitHub repo、本機 git 狀態、每個 agent spawn 的完整執行軌跡（thinking、tool calls、structured output、cost） 整合成一個即時觀測工作站。dispatch 決策即時推送；call chain 可逐步展開。

dispatch log call chain live cost blocking graph

四角色 · 分工

rlm-modeler
唯一 RLM writer。從 intake / 逆向工程 / 執行反饋寫入 project-rlm/。任何模型變動都走這個角色。
arch-shape
讀 RLM，把商業意圖分解成帶完整 spec + AC + deps 的 atomic 子任務。召喚 advisor 做現況查詢。
advisors
be / fe / ops / design — read-only。回答現況問題（現有 service？衝突元件？blast radius？）不寫任何狀態。
implementer
收 RLM-grounded child issue，按需載入 scenario skills，TDD 產出代碼，開 PR。模式 A/B/C。

孿生接收哪些信號

孿生的全部知識來自三個來源：SQLite（持久化）、Tauri events（即時推送）、GitHub API（15 分鐘同步）。

信號來源 內容 觸發時機 孿生可見度
dispatch_log 每次 issue 被評估的決策：rule index、directive（spawn_fresh / no_action / wait / human_review）、reason、matched_at workflow 引擎每次 poll 到 status:ready issue 即時可見
one_shot_runs run metadata：role、argv（含 --name 帶 role+issue）、status、cost、started/ended agent spawn 後立刻寫入 + 推送 oneshot:created event 即時可見
run_events 結構化事件序列：thinking / tool_use / tool_result / result，含 input_json / output_json / is_error run 結束後從 stream-json log 解析，lazy-parse on demand call chain 可展開
one_shot_log_lines 原始 stdout/stderr 逐行，stream-json NDJSON oneshot:line:{id} 事件逐行即時推送 即時串流
graph_edges issue → issue 「blocked_by」邊（從 <!-- deps: --> 解析）；issue → run「dispatched_to」邊 workflow 引擎解析 deps 後寫入 blocking graph 可視
GitHub API cache issues（title / labels）、PRs（title / draft / merged） fetch_issues / fetch_prs，15 min 自動刷新 15 min 延遲
pre_spawn 結果 Abort / Reroute / Continue 決策及原因 runtime 執行但未記錄到任何持久化表 暗區 · 不可見
on_result 步驟 handler 執行了哪些 steps（CreateIssue / Comment / Label / push_branch_and_pr / run_command） runtime 執行但未記錄 暗區 · 不可見

主流程 · 孿生視角 — 11 個階段，每個階段孿生看到什麼

0
起點：空專案 git state

新建 git repo，可能只有空 README 或 .gitignore。沒有 project-rlm/，沒有業務代碼，沒有 GitHub issues。

孿生觀測
  • HomeView 顯示 repo 基本資訊（本機路徑、default branch、git clean 狀態）
  • Dispatch Log 區塊：無記錄 — "No dispatch decisions recorded yet."
  • Blocking graph：無依賴
  • One-shot sidebar：無 run
1
商模意圖進場 issue #1 created

使用者開了第一個 issue 描述商業需求，body 帶 <!-- intake-kind: business -->，labels 加上 status:ready。

孿生觀測
  • HomeView Open Issues 列表出現 #1（15 min 內或手動 Refresh 後）
  • Labels 顯示 status:ready 顏色徽章
  • Dispatch 尚未發生，dispatch_log 還沒有記錄
2
workflow 偵測 + 路由決策 agent-sweet-home poll

workflow runtime 讀 agent-team-v2.workflow.yaml，評估 dispatch rules：

規則評估（依序）
  • project-rlm/ 不存在 + intake-kind: business → spawn_fresh role: rlm-modeler
  • 規則命中後寫入 dispatch_log，emit dispatch:logged event
孿生觀測
  • 即時：dispatch:logged event 觸發，HomeView Dispatch Log 新增一行
  • 顯示：spawn #1 · rule 0 · role: rlm-modeler · 時間戳
  • One-shot sidebar 出現新 run（oneshot:created event）
3
rlm-modeler bootstrap rlm-modeler

從 business intake 推導出初始 RLM 模型，是唯一 RLM writer 第一次落筆。

執行（透過 rlm CLI）
  • rlm define-brief — 產品 brief
  • rlm define-context — 切 bounded contexts
  • rlm define-glossary — 領域詞彙
  • rlm define-story — 用戶故事
  • rlm validate — 強制一致性驗證

產出 project-rlm/ PR 對應 #1，打上 arch-state:awaiting-bootstrap-merge

孿生觀測
  • One-shot chat view：即時串流 thinking blocks（折疊，顯示字元數）
  • Tool call chain 逐步展開：每次 Bash / Read / Write 都可看到 input/output
  • Run header：role badge rlm-modeler + issue #1 + 即時燒錢速率
  • Run 完成後：Structured Output 展開 JSON（含 agent 回傳的 kind 欄位）
  • pre_spawn hook 評估結果不可見（runtime 未記錄）
⚠️ 人工 gate — bootstrap PR 必須人工審查並 merge。這是 RLM 第一次定型，需要人類確認方向。孿生在 HomeView PRs 列表看到這個 PR，但 merge 決策在人類手上。
4
PR merged → workflow 重新評估 #1 PR merged

Trigger：PR closed（merged）且 #1 仍 open。

  • project-rlm/ 存在 → ✓；有業務代碼 → ✗；有子 issues → ✗
  • → dispatch rules → spawn_fresh role: arch-shape
孿生觀測
  • Dispatch Log 新增一行：arch spawn · #1 · 新 rule index
  • Sidebar One-shot 出現第二個 run，role badge = arch-shape
5
arch-shape 分解子任務 arch-shape

讀 RLM 完整模型，根據 contexts + stories 切出可獨立執行的子任務，一次性輸出完整依賴圖。

產出（atomic）
  • #2 subdomain: scaffold — deps: []
  • #3 subdomain: feature-A — deps: [#2]
  • #4 subdomain: feature-B — deps: [#2, #3]
  • 每個 issue body 帶完整 spec + AC + RLM context slug
孿生觀測
  • arch-shape run call chain：可見它用哪些工具讀 RLM（File, Bash）、如何組裝 spec
  • Structured Output JSON 含 arch-shape 產出的子任務清單
  • HomeView Issues：#2–#N 出現在 Open Issues 列表（15 min 後或手動刷新）
  • Blocking graph：deps 關係寫入 graph_edges → 立刻顯示 blocked issues
6
workflow 派發子任務 dispatch cascade

workflow 根據 deps 計算可執行性，無依賴的子任務可並行執行。

  • #2 deps 為空 → status:ready → spawn executor
  • #3 #4 等待 #2 → status:blocked
孿生觀測
  • Dispatch Log 出現多行：每個子 issue 各一條 dispatch 決策
  • #2 → spawn；#3 #4 → wait（顯示 reason）
  • Blocking graph 顯示 #3 #4 被 #2 block
  • One-shot sidebar：新增 executor run，role badge = worker
7
implementer 拾取技能 + 實作 implementer

每個 executor spawn 收到一個 RLM-grounded child issue，按需載入 scenario skills，TDD red-green-refactor。

spawn 輸入
  • subdomain、spec（RLM story 衍生）、acceptance_criteria、deps
scenario skills（按需）
  • API endpoint → api-design + tdd
  • UI 元件 → frontend-ui + a11y
  • schema → migration-safety
孿生觀測
  • 即時串流：One-shot chat view 顯示每一個 tool call（Bash, Edit, Read, Write…）
  • Thinking blocks 折疊顯示，字元數可見，展開讀完整推理過程
  • Tool input / output 可展開（TDD 跑了哪些測試？lint 結果如何？）
  • Run header：issue #N badge + worker role + 即時 cost 累積
  • pre_spawn Abort/Reroute 不可見；on_result steps 不可見
8
PR 開出 → review → merge → issue close PR lifecycle

implementer 發 PR，另一個 spawn 可進入 review mode。CI 通過 → 合併 → issue 自動關閉。

孿生觀測
  • Run call chain 底部 result bar：completed + 總 cost
  • Structured Output 展開：agent 回傳的完整 JSON（PR branch、structured kind、summary）
  • HomeView PRs：新 PR 出現（15 min 或手動刷新）
  • push_branch_and_pr 是 on_result step，成功/失敗記錄不可見
9
workflow 解鎖下游 unblock pass

Trigger：issue #2 closed。unblock pass 掃描所有 status:blocked，重新計算 deps，解鎖就緒的下游。

孿生觀測
  • Dispatch Log 出現新批次：#3 → spawn，#4 → wait（#3 未完成）
  • Blocking graph 自動更新：#3 從 blocked 移除
  • unblock pass 本身的執行細節不可見（無事件記錄）
  • on_unblock steps 執行情況不可見
🔁 主迴圈 — Stages 6–9 對每個子任務循環，直到全部 closed
10
arch-shape 收尾驗收 arch-shape

Trigger：最後一個子 issue closed，parent #1 仍 open。arch-shape 確認原始商業意圖是否達成，視情況派 rlm-modeler 補寫模型，最後關閉 #1。

孿生觀測
  • arch-shape run call chain：可見它讀了哪些 RLM 文件、哪些 PR diff
  • Dispatch Log 出現 arch-shape 的最後一次 spawn 記錄
  • 若需 RLM 更新：rlm-modeler 再次出現在 sidebar，role badge 切回黃色
11
商模需求完成 done RLM ✓

系統終態：project-rlm/ 反映完整領域模型；業務代碼可測試可部署；每個改動可追溯到 RLM story；成本完整記帳。

孿生觀測全貌
  • Dispatch Log：完整的 N 次評估記錄，涵蓋每個 issue 被派發的決策歷程
  • One-shot sidebar：所有 run 列表，role badge 顯示誰做了什麼，cost 全部加總可見
  • Call chain：每個 agent 的 tool call 序列可回放，thinking 可逐一讀
  • HomeView：Issues 和 PRs 列表回到空白，Blocking graph 清零

觀測覆蓋率 · 哪裡有光，哪裡還暗

綠色 = 孿生可完整觀測；橙色 = 部分可見；灰色 = 暗區，runtime 執行但不留任何可查詢記錄。

Dispatch 決策
dispatch_log 記錄每次 rule 評估結果：哪條規則、什麼 directive、reason、issue number、時間戳。即時推送到 HomeView。
Spawn 執行
one_shot_runs 表：role、prompt、argv、status、cost、duration。sidebar 即時更新，call chain 可展開。
Tool Call 序列
run_events 從 stream-json 解析，包含 tool_use / tool_result 配對、is_error、input/output JSON。可展開讀完整 I/O。
Thinking Blocks
Extended thinking 的內容完整保留在 run_events，chat view 顯示字元數 + 可折疊展開讀完整推理。
Structured Output
result event 的完整 output_json（含 kind + 所有 agent 決策欄位）在 call chain 底部展開顯示。
Issue Blocking Graph
graph_edges 的 blocked_by 關係，HomeView 顯示哪些 issue 被哪些 deps 擋著，即時更新。
Issue / PR 列表
GitHub API 快取，15 分鐘自動刷新。Labels 即時顯示（workflow 加的 vs. 人工加的無法區分）。
Cost 歸因
每個 run 的 total_cost_usd 可見。但 per-tool-call cost breakdown 不可見，phase 間的成本分配需手動加總。
Agent Role
從 dispatch_log.directive_json.role 或 argv --name flag 反推，新舊 run 都可識別。mode 資訊部分可見。
Pre-Spawn 結果
apply_pre_spawn() 的 Abort / Reroute / Continue 結果不記錄。spawn 中止時孿生完全不知道為什麼。
On-Result Steps
handler 執行了哪些 steps（CreateIssue / AddLabels / push_branch_and_pr / run_command）沒有任何記錄。成功失敗都不知道。
Error Quarantine
dispatch 或 spawn 出錯後 runtime 加 human-review label，無法區分「設計上的人工審查」和「錯誤後的隔離」。

情況變體 — 主流程的常見偏離 · 8 種

V1 RLM 已存在，新商業需求
第二個以後的商業 issue。最常見。
跳過 bootstrap → workflow 偵測 project-rlm/ 已存在 → 直接派 arch-shape Stage 5。若新需求引入新 context，arch-shape 發 rlm_changes_needed 先讓 rlm-modeler 擴充模型。
Dispatch Log 第一條即為 arch-shape spawn，role badge 直接跳到藍色。rlm-modeler run 只在有模型更新時出現。
V2 既有代碼導入
repo 已有業務代碼但 project-rlm/ 不存在。
rlm-modeler 進入 reverse-bootstrap：讀代碼結構逆向推導 contexts / stories → PR → 人工確認後進主流程。
Call chain 中可見大量 Read / Bash tool calls（rlm-modeler 在掃代碼結構）。Thinking blocks 顯示它如何從代碼推導 bounded context。
V3 arch-shape 需要 advisor 諮詢
arch-shape 不確定現況（"現有類似 service 嗎？"）。
arch-shape 標 arch-state:awaiting-consultation → workflow 派 advisor（be/fe/ops/design）→ advisor 留結構化 comment → arch-shape 重啟。
Dispatch Log 出現 advisor spawn。Sidebar 顯示 advisor run，role badge = advisor（green）。arch-shape 重啟時出現第二個 arch run。
V4 架構決策需要 ADR
arch-shape 遇到架構級選擇。
arch-shape 草擬 ADR → 人工 gate 選方案 → arch-shape 重啟把 implements-adr 寫入子 issue → Stage 6。ADR 由 rlm-modeler 寫入 project-rlm/decisions/。
arch-shape run 結束但 Dispatch Log 無後續 spawn，說明在等人工決策。ADR merge 後新 arch-shape spawn 出現。
V5 implementer 反推（Mode C）
spec 矛盾 / 缺資訊 / 違反 RLM。
executor 留 pushback comment → 標 mode-c-rounds:1 → arch-shape 重新檢視 → 修正 spec → 派回 executor。達到 rounds:3 升級為 human-review。
Dispatch Log 同一 issue 出現多次 spawn 記錄。Call chain 中 structured output 的 kind 欄位為 pushback / mode-c。同一 issue 重複出現的 executor run 可見。
V6 executor 發現 RLM 漂移
實作中發現 RLM 與現實領域不符。
executor 不越權 → 在子 issue 留 rlm_changes_needed → workflow 派 rlm-modeler 修正 → RLM PR merge → executor 重啟。
Sidebar 出現 rlm-modeler run（黃色 badge）夾在兩個 executor run 之間。Dispatch Log 顯示完整路徑：executor spawn → wait → rlm-modeler spawn → executor re-spawn。
V7 小型 bug fix 直接派工
issue 範圍小、不影響領域模型。
dispatcher 判斷無商業意圖 → 跳過 arch-shape → 直接派 executor → PR → merge。
Dispatch Log 只有一條 executor spawn，無 arch-shape 記錄。整個流程 Dispatch Log 只有一行，call chain 短而直接。
V8 arch-judgment 兜底
dispatcher 5 條規則都不命中 / agents 衝突 / malformed intake。
派 arch-judgment（escape-hatch 變體）→ 讀更廣 context → 做一個有界決策 → 退出。預期觸發率 1–5%。
Dispatch Log 出現 arch-judgment 的特殊 role badge。Call chain 中可見它讀了 RLM survey + git history + parent thread 等更廣的 context。

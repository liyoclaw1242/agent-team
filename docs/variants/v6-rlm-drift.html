V6 · 模型校準

executor 發現 RLM 與現實不符

executor 實作中發現 RLM 描述的 entity / story 與真實領域有出入。executor 不偷改 RLM (沒寫權)，而是發信號讓 rlm-modeler 修正。

觸發條件
executor 實作中發現 RLM 不準確 / 缺漏 / 概念混淆
主要角色
executor (發信號) · rlm-modeler (修正) · arch-shape (重評估)
偏離主流程
主流程 Stage 7 暫停 → 修 RLM → 可能重切子任務 → 重啟 Stage 7

前置狀態

executor 在 Stage 7 拿到子 issue，spec 看起來合理 (沒觸發 V5 pushback)，開始實作。但深入後發現 RLM 描述與真實領域有出入：

  • 缺欄位：RLM 說 Order entity 有 customer_id，但業務需要 billing_address + shipping_address 多個地址
  • 概念混淆：RLM glossary 把「會員」和「使用者」當同義，實作時發現是不同概念 (員工內部使用 vs 顧客)
  • 狀態機缺失：RLM story 沒提到「訂單退款後可以重新下單」這個分支
  • 邊界錯位：RLM 把 inventory 放在 orders context，但實作上需要獨立 context (多個 channel 共用)
  • 事實已過時：RLM 說「使用 Stripe」，但團隊已改用 Adyen，文件未更新

觸發條件

executor 在實作中執行 RLM-grounded 比對。任一條件成立 = V6：

1. spec 引用的 RLM entity 欄位與現實 schema 不同？
2. RLM glossary 詞彙在代碼裡有多種用法 (語意不一)？
3. spec 描述的 user flow 與測試 / UI 反映的真實 flow 不同？
4. 為了完成 acceptance criteria，必須引入 RLM 沒登錄的概念？
5. 現有代碼已採用某 pattern，但 RLM 描述的是另一種？

任一 yes → emit rlm_changes_needed 信號，executor 暫停

詳細流程

executor 開始實作 executor

spec 通過 sanity check (不是 V5)，executor 進入實作階段。

例：spec 是「為信眾資料新增 CRUD」，引用 RLM 的 Member entity。

發現 RLM 與現實落差 executor

實作過程中，executor 對照 RLM 與代碼/業務需求發現問題。

具體例
RLM `payments.Member` 描述：
  fields: id, name, phone, email

但實作時發現業務需要：
  - 法名 (vs 俗名) — 法會報名時用法名
  - 戶籍地址 vs 通訊地址 — 開立功德證明用前者
  - 緊急聯絡人 — 法會意外處理用

這些不是「我想多加幾個欄位」— 而是 RLM 漏了真實的領域概念。
executor 暫停 + 發信號 executor

executor 不偷改 RLM (沒有寫權)，不硬實作多餘欄位 (會與 RLM 漂移)。

留 comment 在子 issue：

<!-- rlm_changes_needed -->
trigger: executor mid-implementation
issue: spec 引用 RLM `payments.Member`，但業務需求需要 RLM 沒記錄的欄位

evidence:
  - acceptance criteria #3 提到「列印功德證明」需要戶籍地址
  - 既有測試 (tests/legacy/funeral.spec.ts) 用了 dharma_name 欄位
  - 與 #N 父 issue 的 stakeholder 訪談紀錄有提到緊急聯絡人

proposed RLM changes:
  - 擴充 Member entity，新增：
    - dharma_name (法名，optional)
    - registered_address (戶籍地址)
    - mailing_address (通訊地址，可空)
    - emergency_contact (json)
  - 新增 glossary：法名 vs 俗名、戶籍 vs 通訊
  - story 補充：「列印功德證明」流程

executor_status: paused (waiting for RLM update)

labels: arch-state:awaiting-rlm-update status:blocked

workflow 派 rlm-modeler dispatcher rlm-modeler
trigger子 issue 有 rlm_changes_needed 信號

workflow 開新的 sub-issue 給 rlm-modeler，描述要做的 RLM 變更。

rlm-modeler 不是盲目套用 executor 提案 — 它會：

  • 讀 executor 的 evidence 重新評估提議的合理性
  • 檢查是否影響其他 context (e.g. dharma_name 是否其他 service 也需要？)
  • 必要時拒絕提議或修改範圍 (e.g. 「emergency_contact 應該獨立 entity，不該塞 Member」)
rlm-modeler 修正 RLM rlm-modeler
執行
  • rlm update-entity Member --add-field=dharma_name --type=string?
  • rlm define-glossary 法名 --description=...
  • rlm define-glossary 戶籍地址 --description=...
  • rlm update-story member-registration --add-step=...
  • rlm validate

產出 PR：「fix: align RLM Member entity with funeral domain reality (#X)」

RLM PR review + merge PR

review 重點：

  • 新欄位是否真的屬於 Member 還是該獨立
  • glossary 是否與既有詞衝突
  • story 變更是否影響其他子任務

通常這類 PR 比 bootstrap 簡單 (是局部修正)，可較快合併。

arch-shape 重評估子 spec arch-shape
triggerRLM PR merged + 原子 issue 標 awaiting-rlm-update

workflow 派 arch-shape (而非直接派 executor) 因為 RLM 變了，子 issue 的 spec 可能也要更新。

arch-shape 三種可能動作：

  • 更新 spec：寫進新欄位 / glossary / step → 派回 executor
  • 拆子任務：發現原本一個 issue 該拆兩個 (e.g. 「Member CRUD」+「emergency_contact 管理」)
  • 連鎖檢查：是否有其他子 issue 也受影響？(e.g. 法會報名 issue 也會用到 dharma_name)
executor 重啟實作 executor

spec 已對齊新 RLM，executor 進入正常實作流程。

後續同主流程 Stage 8 (PR / merge / 關閉)。

分支與失敗模式

狀況後果處理
executor 提議的 RLM 變更實際上是業務變化 (新需求) 不該偷渡進現有 issue rlm-modeler 拒絕，改為要求人類開新 issue 走主流程
RLM 修正影響大量現有子 issue (連鎖效應) 多個 issue 需重評估 arch-shape 對所有受影響 issue 標 awaiting-rlm-realign，批次處理
同一 entity 在短時間內被多個 V6 修正 RLM 翻來覆去 升級為 V4 — 該 entity 的設計需要 ADR，不該邊做邊調
executor 「假報」V6 為了避開難實作的 spec 系統被濫用 rlm-modeler review 時要求 evidence — 沒有 evidence 的 V6 應視為 V5
RLM 修正後執行還是失敗 原因不是 RLM 進入 V5 (Mode C pushback)

與主流程的接點

RLM 校準完 → arch-shape 修 spec → executor 重啟。後續流程 (Stage 8-11) 不變。

V6 留下的痕跡 (RLM PR + executor pushback comment) 是系統健康度的指標：

  • 同一 context 的 V6 頻率高 = bootstrap 階段建模草率
  • 同一 entity 反覆 V6 = 該 entity 應升級為 V4 ADR
  • V6 集中在某個 advisor 沒參與的 subdomain = 該 advisor 應更早介入 (V3)

為何這樣設計

權限分離保護模型完整性　executor 沒有 RLM 寫權是關鍵紀律。如果 executor 能直接改 RLM，每個 spawn 都會「順手」加自己需要的欄位，幾週後 RLM 變成所有 executor 改動的雜燴，失去單一來源的價值。

「暫停而非繞過」的紀律：

  • executor 寧可不開 PR 也不寫 RLM 沒記錄的概念 — 因為下次 advisor 查 RLM 看不到，會誤判現況
  • 強制走 RLM PR，意味著 RLM 變更被獨立 review — 而不是混在「順便加幾個欄位」的 code PR 裡偷渡
  • 每個 V6 都是 RLM 與現實對齊的機會；這是系統自我修復的機制

相關變體

V5 · Mode C pushback 當問題是 spec 寫錯而非 RLM 不對，應走 V5 V1 · RLM 已存在 V6 修正後續的需求都走 V1，引用更新後的 RLM V4 · ADR 人工 gate 同一 entity 反覆 V6 應升級為 ADR V3 · advisor 諮詢 advisor 在諮詢時也可能發現 RLM 不準，發 V6 信號
V6 是「模型校準」的常規迴路。健康的專案會週期性觸發 — RLM 從來不會一次寫對，只能透過實作不斷對齊現實。

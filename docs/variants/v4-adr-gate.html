V4 · 重大決策

架構決策需要 ADR

arch-shape 遇到「在多個合理選項間選擇」時暫停，草擬 ADR，等人工選定方案後再繼續分解。

觸發條件
arch-shape 識別出 ADR-worthy 的設計選擇
主要角色
arch-shape (草擬) · rlm-modeler (寫入) · 人類 (選擇)
偏離主流程
在 Stage 5 中段插入 ADR draft → 人工 promote → 重啟分解

前置狀態

已在主流程 Stage 5 (arch-shape 分解)。arch-shape 發現某個子任務的實作方式有多個合理選項，每個選項都有實質的 tradeoff，而且選擇會影響後續多個子任務 — 這就是 ADR-worthy。

典型 ADR 觸發場景：

  • 技術棧選擇 (Postgres vs MongoDB / REST vs GraphQL / SSR vs CSR)
  • 跨 context 整合方式 (同步呼叫 vs event-driven / monorepo vs polyrepo)
  • 資料一致性策略 (強一致 vs 最終一致)
  • 身份驗證模型 (session vs JWT / 自建 vs 第三方)
  • 部署拓樸 (單一服務 vs 拆分 / serverless vs 容器)

觸發條件

arch-shape 自我判斷時的判準：

檢查是 → V4
是否有 ≥ 2 個合理選項？有
選項之間的 tradeoff 是否實質 (非偏好)？是
選擇是否影響 ≥ 3 個子任務？是
RLM 是否已記錄類似決策？否
選項是否需要組織 / 業務 / 預算考量？否則 advisor 諮詢就夠 (V3)

若 5 條都符合 → arch-shape 草擬 ADR；任一條不符 → 應該走 V3 諮詢或 arch-shape 自決。

詳細流程

arch-shape 識別 ADR-worthy 點 arch-shape

分解時遇到判準命中的選擇點。arch-shape 不假設答案，而是把問題 + 選項結構化。

範例：奉獻金流的同步性
  • 問題：信眾完成奉獻後，發票該同步產生還是非同步？
  • 選項 A：同步 — 付款回應前產生發票，UX 即時但失敗風險高
  • 選項 B：非同步 — 付款後排入 queue，UX 需等通知但更穩定
  • 影響子任務：付款 service / 發票 service / UI 完成頁面 / 通知 worker
arch-shape 草擬 ADR arch-shape

arch-shape 發 rlm_changes_needed 信號，請求把 ADR draft 寫進 RLM。

<!-- rlm_changes_needed -->
- new ADR (draft): 0042-invoice-issuance-sync-vs-async
  context: payments
  status: draft
  question: 發票產生時機 — 同步 vs 非同步
  options:
    A: 同步產生
       pros: UX 即時，邏輯線性
       cons: 第三方稅務 API 失敗會擋付款流程
    B: 非同步產生 (event-driven)
       pros: 解耦，可重試，不擋付款
       cons: UX 需等待通知，需要 worker 基礎設施
  recommendation: B (穩定性優先)
  decision: AWAITING HUMAN

arch-shape 標 arch-state:awaiting-decision-approval 並退出。

rlm-modeler 寫入 ADR draft rlm-modeler

workflow 派 rlm-modeler 把 ADR 寫入 project-rlm/decisions/draft/。

執行
rlm define-decision \
  --slug=0042-invoice-issuance-sync-vs-async \
  --status=draft \
  --context=payments \
  --options=...
rlm validate

產出 PR：「ADR-0042 draft (awaiting decision)」。

PR merged → ADR draft 入庫 PR merged

這個 PR review 重點不是「方案對不對」(那是下一步)，而是「選項列得完整嗎？tradeoff 寫得對嗎？」

merged 後 ADR 進入 decisions/draft/，#N 仍維持 arch-state:awaiting-decision-approval。

⚠️ 人工 gate (核心)　ADR 的選擇必須由人類做，不能交給 LLM。這個 gate 才是 V4 的本質 — agent 系統把問題結構化，人類選方向。

人工 promote 流程

人類 review ADR human

人類在 decisions/draft/0042-... 中做出選擇，可選方式：

  • 選擇某個選項：編輯檔案，把 decision: AWAITING HUMAN 改成 decision: B，補上 rationale
  • 要求重新草擬：留 comment 指出選項列得不夠或缺漏，arch-shape 重做
  • 引入新選項：加 option C，arch-shape 重新評估
  • 拒絕 ADR：判定不需要決策 (e.g. 已有先例)，改走別的路徑
promote draft → accepted human rlm-modeler

人類確定後，透過 rlm-modeler (或 rlm CLI 直接) 把 ADR 從 draft/ 移到 accepted/，狀態改為 accepted，加上 decision date 和 decided-by。

rlm promote-decision 0042-... --status=accepted \
  --decision=B --decided-by=@your-handle

這個動作會發 git commit，trigger workflow 重新檢查 #N。

workflow 重啟 arch-shape dispatcher
triggerADR promoted to accepted + parent #N 仍 open

workflow 移除 awaiting-decision-approval，重派 arch-shape，把 ADR slug 一併傳入。

arch-shape 用 ADR 完成分解 arch-shape

arch-shape 讀 ADR-0042 (option B 已選定) 後切子任務，每個受影響的子 issue body 都標上 implements-adr: 0042：

子 issue #X
<!-- subdomain: payments -->
<!-- implements-adr: 0042-invoice-issuance-sync-vs-async -->
<!-- deps: ... -->

spec: 建立 invoice queue worker，消費 PaymentSucceeded event...

implements-adr 標記讓 executor 在實作時可以追溯到決策原因，避免重新發明。

分支與失敗模式

狀況後果處理
人類遲遲不選 #N 永遠卡在 awaiting-decision-approval 系統不催；列入 stale issue 報表，由人類定期清理
人類選了未列出的方案 (option D) arch-shape draft 沒涵蓋 視為「重新草擬」— rlm-modeler 更新 ADR 加入 D，重做 review
多個 ADR 互相依賴 需要連鎖決策 arch-shape 一次草擬全部，標 blocks: 0043, 0044，人類批次決
已 accepted 的 ADR 後來證明錯誤 需要 supersede 新開 ADR-NNNN，標 supersedes: 0042；舊 ADR 改 status: superseded 但保留
arch-shape 把不該開 ADR 的事情升級成 ADR 過度結構化、拖慢流程 review 時拒絕 draft；arch-shape 改走 V3 諮詢或自決

與主流程的接點

ADR promoted 後，arch-shape 重啟分解 → 接主流程 Stage 5 → 後續 (Stage 6-11) 不變。

每個受影響的子任務 PR 在實作時都會引用 ADR slug；reviewer 看 PR 也能即時對照「為什麼要這樣做」。

為何這樣設計

把選擇還給人類　LLM 可以列選項、估 tradeoff，但不該替組織做架構決策。架構選擇牽涉組織能力、預算、招聘、風險偏好、政治 — 這些是人類的職責。V4 的價值是「把決策結構化」，讓人類更快做決定，而不是繞過人類。

把 ADR 寫進 project-rlm/decisions/ 的好處：

  • 決策成為 RLM 的一部分，未來的 arch-shape 能查到先例
  • 子 issue 的 implements-adr 標記提供反向追溯
  • ADR 修訂歷史 (git log) = 組織決策的演化史
  • 新人 onboarding 看 RLM 就能理解「為什麼是這樣」

對比「在 PR 描述裡寫決策」(常見的反模式)：PR 會被歸檔遺忘，新需求出現時又從頭重論一遍。ADR 是持久的決策原語。

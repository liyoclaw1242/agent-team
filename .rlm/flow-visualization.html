🔑 從零情境的觀察

  1. ADR 是 gate 不是裝飾。Hermes 開了 ADR PR 後必須等 liyo merge 才能 commit 帶 adr_refs 的 WP — rlm approve-workpackage 機械驗證會擋下未 merged 的 ADR ref(per ADR-0013)。這把「先決定再執行」變成可強制的契約。
  2. depends_on 串成 pipeline。Dispatch 看到 #5 #6 #7 的 deps 還沒 delivered 時直接跳過,只挑 #4 進場。Serial worker + dependency-aware queue 自然形成。
  3. Worker retry 是常態,不是異常。#7 的 BlackBox attempt 1 失敗只是 implementation defect(missing middleware),retry 計數標籤 + 失敗回饋足以讓 attempt 2 從 fresh claude -p 修對。Stateless 不等於沒記憶 — 記憶在 Issue / branch / event log。
  4. Spec 永遠停在 confirmed。Spec 是「當時要解的問題」,code 上線後 Spec 不會變成 delivered。「當前狀態」永遠是 main + .rlm/facts/。這避免 SDD 走幾輪後「九個月前的 Spec 算不算現在的事實」這種崩塌。
  5. 從 liyo 視角還是同一條 thread。從「想做 todo list」到「上線 4h 40min」全在「household-todo」這一個 thread 跑完。Hermes 切 skill、Dispatch / Worker / Validator 在背景跑、ADR / WP 走 GitHub — 但跟人類的接觸面只有一個 Discord thread + GitHub PR review 介面。

V8 · 兜底

arch-judgment 兜底

dispatcher 5 條規則都不命中、agents 之間衝突、malformed intake — 派 arch-judgment 讀更多 context 做一個有界決策然後退出。預期 1-5% 觸發。

觸發條件
dispatcher 規則 1-4 都未命中 → 規則 5 兜底
主要角色
arch-judgment (是 arch-shape 的 escape-hatch 變體，不是第 5 個概念角色)
預期觸發率
1-5%；高於這個值代表上游規則需要調整

前置狀態

workflow dispatcher 嘗試對某個 issue / state 套用規則 1-4，但都不命中。系統處於模糊狀態：

  • issue body 結構錯誤、缺 marker、或 intake-kind 不認識
  • 多個 agents 留下衝突結論 (e.g. advisor 說 X，arch-shape 已在 spec 寫 Y)
  • parent 已關但 children 仍 open
  • round-3 卡關 (V5 升級點)
  • 子 issue 的 deps 指向已關閉但被 reopen 的 issue
  • 同一 issue 被多個 agent label 同時指定 (label 衝突)
  • RLM 在處理過程中被外部修改 (非 rlm-modeler 來源)

觸發條件 (規則 5)

dispatcher 規則表結構大致如下：

規則 1: 空 RLM + business intake → rlm-modeler bootstrap
規則 2: 有 RLM + business intake → arch-shape decomposition (V1)
規則 3: bugfix/chore + 範圍小 → executor direct (V7)
規則 4: pushback / consultation / awaiting-* signal → 對應 agent
規則 5: 以上都不命中 → arch-judgment (V8)

關鍵設計：規則 5 是顯式的兜底，不是「不知道怎麼辦就丟回去」。arch-judgment 收到的不是無腦轉發，而是一個明確標記為「需要更廣判斷」的 case。

詳細流程

workflow 落到規則 5 dispatcher

規則 1-4 都不命中，dispatcher 標記 issue：

labels: agent:arch-judgment flow:escape-hatch

flow:escape-hatch label 是事後審計用 — 蒐集所有 V8 case 拿來檢視 dispatcher 規則是否需要補強。

arch-judgment 啟動 arch-judgment

arch-judgment 與 arch-shape 共用大部分紀律 (RLM-first / stateless / 不寫代碼)，但 prompt budget 較大，被授權讀更廣的 context：

擴大讀取範圍
  • RLM 完整 survey (不只當前 context)
  • git history (recent 50 commits)
  • parent issue 完整 thread (含所有 comment)
  • 相關子 issue 的 comment
  • 衝突 agents 的留言原文
  • 近期同類 issue 的處理結果 (作為先例)

這些讀取量在 arch-shape 是不允許的 (token 浪費)，但在 V8 是必要的 — 因為 V8 case 本身就需要更多 context 才能判斷。

做一個有界決策 arch-judgment

arch-judgment 不解決問題，它路由問題。決策必須是下列其一：

允許的決策類型
  • 路由到正確 agent：「這其實是 V3，派 advisor」「這是 V4，需要 ADR」
  • 修補 label / 標記：「補上 intake-kind: business，重新 dispatch」
  • 升級 human-review：「這超出系統能處理的範圍，請人類介入」
  • 關閉冗餘 issue：「這是 #N 的重複 / 過時，關閉」
  • 合併或拆分：「兩個 issue 應該合併」「一個 issue 應該拆成三個」
  • 拒絕：「這個 case 不該被處理」(留 comment 說明)
寫 reasoning comment 然後退出 arch-judgment

無論決策是什麼，arch-judgment 都必須留下完整 reasoning：

<!-- arch-judgment -->
case_signature: round-3 escalation, conflicting verdicts
context_examined:
  - parent #42 thread (12 comments)
  - executor pushback at #45, #45 (2 rounds)
  - advisor-be consultation at #46
  - RLM survey: payments + billing contexts
findings:
  - executor 與 arch-shape 卡在 "同步 vs 非同步" 的選擇
  - 兩輪 pushback 都圍繞同一 tradeoff
  - 這是 ADR-worthy 問題，不是 spec 表達問題
decision: route to V4 (ADR drafting)
action: |
  - 移除 mode-c-rounds:N labels
  - 標 arch-state:awaiting-decision-approval
  - 重派 arch-shape (with hint: "draft ADR for sync-vs-async")
exit: routed

arch-judgment 然後退出 — 它不執行後續路由結果 (派 arch-shape 是 workflow 的工作)。

workflow 套用 arch-judgment 的決策 dispatcher

workflow 讀 arch-judgment 的 action 段，照著做：

  • flip labels
  • 派下一個 agent (如指定)
  • 關閉 issue / 開新 issue (如指定)

arch-judgment 的決策一次性 — 同一 issue 不會在短時間內被 V8 處理兩次 (avoid loop)。

分支與失敗模式

狀況後果處理
arch-judgment 也判不出來 沒有可用決策 強制升級 human-review，留下 reasoning 等人類介入
arch-judgment 路由錯誤，下游又卡住 同 issue 再次落到 V8 系統限制：同一 issue 24h 內不會 V8 兩次。第二次直接升級 human-review
V8 觸發率超過 10% dispatcher 規則設計有漏洞 批次分析 flow:escape-hatch issue，找出 pattern，新增規則
arch-judgment 想做的不是路由 (e.g. 想直接寫 RLM) 違反角色紀律 spawn 強制限制：可用工具不含 Edit/Write 對 RLM；只能輸出 routing decision
多個 issue 同時觸發 V8 (系統性問題) 不是個別 case，是規則漏掉某類場景 暫停自動 dispatch，由人類批次處理 + 修規則

與主流程的接點

V8 不接主流程 — 它做出路由決策後，issue 進入決策指定的下一個變體 (V1/V3/V4/V7) 或主流程，或直接到 human-review 終結。

V8 的事後價值：每一次 V8 觸發都是 dispatcher 規則的失誤。批次審 V8 case = 找出規則漏洞 = 系統演化。

為何這樣設計 (V8 的存在意義)

明確的「我不知道怎麼分」　沒有 V8 的話，模糊 case 會隱身在某個錯誤的規則裡 (例如被誤判為 V7 直派 executor)，造成靜默故障。V8 把不確定性顯式化 — 讓「不知道怎麼處理」變成可審計的事件。

V8 的設計約束：

  • 稀少使用：1-5% 觸發率是設計目標。太高 = 規則太鬆 / 太嚴；太低可能是 V8 觸發條件設計太嚴 (規則應該命中卻沒命中的 case)
  • 一個決策：arch-judgment 不迭代、不嘗試多個方案 — 一次 spawn = 一個決策 = 退出
  • 可審計：所有 V8 case 都帶 flow:escape-hatch label，便於批量分析
  • 不執行：arch-judgment 只決定「該由誰處理」，實際處理交給對應的 agent

為何不直接派人類

跳過 arch-judgment 直接 escalate human-review 看似更穩。但實務上：

  • 大多數 V8 case 是路由問題而非決策問題 — 系統能判，只是規則沒涵蓋
  • 每個 V8 都打擾人類會讓人類疲乏 (見了狼來了多次後不再認真看)
  • arch-judgment 的 reasoning comment 同時是「給人類的 brief」— 即使最後升級人工，人類也省了重新調查的時間

為何不是第 5 個概念角色

arch-judgment 與 arch-shape 共用：

  • 同一 RLM 關係 (讀 RLM，不寫)
  • 同一輸出形式 (路由決策 / 結構化 comment)
  • 同一 stateless 紀律

不同的只是prompt budget (允許讀更多)。所以概念上仍是「arch」這個角色，arch-judgment 是 arch-shape 的變體，不是新角色。系統角色仍是四個。

觀察：V8 觸發率作為健康指標

觸發率診斷
< 1%規則設計可能太嚴 (該 V8 的 case 沒被識別)；定期抽查 issue 處理是否有誤分
1-5%健康 — 設計目標範圍內
5-15%規則覆蓋率不足；批次分析 V8 case 找新規則
> 15%系統性問題；可能 dispatcher 邏輯需要重寫，或新類型的場景出現

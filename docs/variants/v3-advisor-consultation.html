V3 · 諮詢分支

arch-shape 諮詢 advisor

分解時遇到「現況是什麼」的問題，arch-shape 暫停並請 advisor (be / fe / ops / design) 給結構化回答，再用結論繼續分解。

觸發條件
arch-shape 在分解過程中發出 need-consultation 信號
主要角色
arch-shape · advisor (1-N 個 discipline)
偏離主流程
主流程 Stage 5 中段插入 advisor 諮詢迴圈

前置狀態

已進入主流程 Stage 5 (arch-shape 分解)。arch-shape 嘗試切子任務時遇到不確定點，這些不確定不是「設計選擇」(那是 V4 ADR)，而是「現況到底是什麼」：

  • 「我們已經有處理付款的 service 嗎？」(後端現況)
  • 「現有的 Form component 支援這種驗證嗎？」(前端現況)
  • 「Postgres 加這個欄位的 blast radius？」(維運現況)
  • 「這個 UI flow 跟現有設計系統哪些 pattern 衝突？」(設計現況)

觸發條件

arch-shape 在處理過程中可以發出 consultation request。當它判斷自己無法只靠 RLM 回答某個問題時：

arch-shape 在 #N 留 comment：
<!-- need-consultation -->
discipline: be          ← 哪個 advisor
question: 我們是否已有處理 Stripe webhook 的 service？
        如有，是否能擴充支援 Apple Pay？
context: 子任務 "信眾付款回呼處理" 的 spec 取決於這個答案

→ workflow 偵測 need-consultation → 派對應的 advisor

arch-shape 自己標 arch-state:awaiting-consultation，然後退出。它不會卡在那裡等。

詳細流程

arch-shape 識別不確定點 arch-shape

分解過程中，arch-shape 嘗試查 RLM 回答某個問題，但 RLM 不夠 (因為 RLM 描述「應有什麼」，不描述「程式碼裡實際有什麼」)。

什麼算「需要諮詢」
  • ✅ 「現況是什麼」— 該找 advisor
  • ✅ 「現有東西能不能擴充支援 X」— 該找 advisor
  • ❌ 「我們應該選 A 還是 B」— 那是設計決策，走 V4 ADR
  • ❌ 「這個 RLM context 邊界是否合理」— 那是 arch-shape 自己的工作
arch-shape 留 consultation request arch-shape

在 #N 留結構化 comment，標 arch-state:awaiting-consultation，然後退出 (stateless)。

comment 格式
<!-- need-consultation -->
discipline: be
subdomain: payments
question: |
  專案是否已有處理 Stripe webhook 的 service？
  如有，能否擴充支援 Line Pay 的 callback signature？
why: |
  子任務 "處理付款回呼" 的 spec 取決於：
  - 重用現有 webhook receiver 還是新建
  - 多 provider 的路由策略 (單一 endpoint vs 多 endpoint)
context: |
  RLM payments context 描述了業務需求，但沒記錄
  現有實作。需要 be advisor 確認。
workflow 派 advisor dispatcher
triggerparent issue 標 awaiting-consultation + 有 need-consultation comment

workflow 開新的 sub-issue 給 advisor (一個 discipline 對一個 sub-issue，同時 fork 多個 advisor 也可以並行)。

新 issue #N+1
labels: agent:advisor-be, status:ready
body: parent #N 的 consultation request 全文
advisor 執行 RLM-first 調查 advisor-be

advisor 是 read-only。它的工作是讀完現況後留結構化建議，不修任何 state。

RLM-first 查詢順序
  • rlm list-components --context=payments — RLM 是否登錄過 webhook component
  • rlm describe-api payments.webhook-receiver — 已建模的 API 形狀
  • fallback：grep 代碼 (src/payments/webhook, routes/webhook.ts)
  • 確認 RLM 與 code 是否一致
advisor 留結構化回答 advisor-be

advisor 在自己的 sub-issue (#N+1) 留 comment，回答 arch-shape 的問題：

## advisor-be · consultation 回覆

### 問題 1：是否已有 Stripe webhook service？
**有**。位於 `src/payments/webhook-receiver.ts`，
RLM slug `payments.webhook-receiver`。
目前只處理 Stripe，但設計時就抽象了 provider 接口。

### 問題 2：能否擴充支援 Line Pay？
**能**。需要的改動：
- 新增 `LinePayProvider implements PaymentProvider`
- webhook 路由改為 `/webhook/:provider` 風格
- 新增 Line Pay signature 驗證 (HMAC-SHA256)

### 風險點
- Line Pay callback retry 行為與 Stripe 不同
  (Stripe 24h 內重試，Line Pay 立即 3 次)
  → idempotency key 處理需重新檢視

### 建議分子任務
1. provider 抽象化重構 (deps: 無)
2. Line Pay provider 實作 (deps: 1)
3. idempotency 統一處理 (deps: 1)

advisor 關閉自己的 sub-issue (#N+1)，標 status:done。

workflow 重啟 arch-shape dispatcher
triggerconsultation sub-issue closed + parent #N 標 awaiting-consultation

workflow 解除 awaiting 狀態，重派 arch-shape 處理 #N，把 advisor 結論一併傳入 spawn context。

labels: agent:arch-shape + 移除 awaiting-consultation

arch-shape 用結論完成分解 arch-shape

arch-shape 收到 advisor 給的具體建議後，產出更精確的子任務：

  • #X provider 抽象重構 deps: []
  • #Y Line Pay provider deps: [#X]
  • #Z idempotency 處理 deps: [#X]

每個子任務的 spec 都引用 advisor comment 連結，executor 之後可追溯。

分支與失敗模式

狀況後果處理
arch-shape 一次需要多個 discipline 的 advisor 需要並行多個諮詢 workflow 開多個 sub-issue (parallel)，arch-shape 等所有都 closed 才重啟
advisor 給的回答含糊或自相矛盾 arch-shape 仍然無法分解 arch-shape 可發第二輪 consultation；超過 2 輪同一問題 → 升級 V8 (arch-judgment)
advisor 發現現況本身有問題 (e.g. 代碼與 RLM 不符) 應該優先修正 RLM advisor 在 comment 標 rlm_changes_needed → 觸發 V6
advisor 自己也不確定 advisor 不該猜 advisor 應坦白「無法判定」並建議升級 (人工 review 或 ADR)

與主流程的接點

諮詢循環在主流程 Stage 5 內部 — Stage 5 結束後 (子任務開出來)，後續流程 (Stage 6-11) 完全相同。

從外部看，#N 多花了一些時間 + 多了若干 closed 的 advisor sub-issue 作為審查痕跡，但最終的子任務品質明顯提升。

為何這樣設計

分工而非合併　不讓 arch-shape 自己 grep 代碼是刻意的。arch-shape 的工作是「以 RLM 為基礎切任務」；要 grep 代碼分析現況是 advisor 的工作。讓 arch-shape 跨界做兩件事會稀釋它的紀律 (RLM-first)。

把 consultation 做成獨立 sub-issue而非「arch-shape 內部呼叫」的好處：

  • 每個諮詢都留下可審查的 GitHub 痕跡 (誰問了什麼、advisor 回答了什麼)
  • arch-shape 是 stateless 的，不需要保留 「我之前問過...」的記憶
  • 同一個諮詢結論可以被後續的 V1 / V5 重新引用
  • advisor 可以並行處理多個 question，不卡 arch-shape

相關變體

V4 · ADR 人工 gate 諮詢之後仍是「需要選擇方案」，升級為 ADR V6 · RLM 與現實不符 advisor 諮詢時最常觸發的副作用 V8 · arch-judgment 同一問題反覆諮詢仍無解時的兜底
V3 是「arch 與 advisor 分工合作」最頻繁的場景。多輪諮詢頻率高 = arch-shape 對 RLM 訊息密度的依賴超過模型實際提供的細節。

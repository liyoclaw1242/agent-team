V1 · 最常見

RLM 已存在，新需求進場

第二個以後的商業 issue 進場時，跳過 bootstrap，直接由 arch-shape 處理 — 但需要判斷是否引入新 context。

觸發條件
project-rlm/ 存在 + 新 intake-kind: business issue
主要角色
arch-shape · 視情況 rlm-modeler
偏離主流程
跳過 Stage 2-4 (bootstrap)；可能在 Stage 5 前插入 RLM 擴充

前置狀態

這個變體假設專案已經至少完成過一輪主流程，project-rlm/ 已建立並 merge：

  • 有完整的 brief / contexts / glossary / stories
  • 可能已有若干 ADR (decisions/)
  • 已有業務代碼，並通過先前商模需求驗收
  • 沒有未完成的 parent issue 卡在 arch-state:tracking-children

觸發條件workflow 如何辨識

新 issue 進場時，dispatcher 依序檢查：

1. issue body 是否含 <!-- intake-kind: business -->     → 是
2. project-rlm/ 是否存在                                  → 是
3. 是否已有 agent:<role> label                          → 否
4. 沒有 deps 阻塞                                         → 否
→ 規則命中：派 arch-shape (decomposition mode，非 bootstrap)

注意這裡不是 Stage 2 的「派 rlm-modeler bootstrap」，因為 RLM 已經存在；workflow 直接跳到「主流程 Stage 5：arch-shape 分解」。

詳細流程

新商業 issue 進場 issue #N

使用者開新需求 issue：

issue body
<!-- intake-kind: business -->
「新增線上奉獻金流：信眾可選擇 Apple Pay / Line Pay / 信用卡⋯」

labels: status:ready

workflow 路由 → arch-shape dispatcher
trigger新 business issue + RLM 已存在

跳過 bootstrap，直接派 arch-shape decomposition mode。

labels: agent:arch-shape status:in-progress

arch-shape 評估 RLM 充分性 arch-shape

讀現有 RLM，判斷新需求所需的概念是否已被覆蓋：

RLM-first 查詢
  • rlm list-contexts — 是否有「金流」或「奉獻」context？
  • rlm list-glossary — 「奉獻」「捐款」「功德金」這些詞已定義？
  • rlm list-stories --context=信眾管理 — 是否已有付款相關 story？

兩種結果，分支處理 →

分支判斷

A RLM 已涵蓋新概念

新需求只是現有 context 的延伸 (例如 brief 裡已寫過「線上付款是 Phase 2」)。

動作：直接進入分解，產出子 issue。等於進入主流程 Stage 5。

B 需要擴充 RLM

新需求引入新 entity / story / context (例如「奉獻」是過去沒有的 bounded context)。

動作：arch-shape 暫停，發 rlm_changes_needed 信號，等 rlm-modeler 補完模型。

分支 B 詳細流程需要擴充 RLM 時

arch-shape 發 rlm_changes_needed 信號 arch-shape

arch-shape 在 #N 留 comment：

<!-- rlm_changes_needed -->
- new context: payments (信眾付款)
  parent: 信眾管理
  reason: 引入 Apple Pay / Line Pay 整合，需獨立邊界處理 PCI 合規
- new glossary: 奉獻 vs 捐款
  reason: 法律語境不同，會影響稅務出據

labels: arch-state:awaiting-rlm-update

workflow 派 rlm-modeler 擴充 dispatcher rlm-modeler

workflow 偵測 rlm_changes_needed 信號 → 開新的 sub-issue 給 rlm-modeler，標 agent:rlm-modeler。

執行
  • rlm define-context payments --parent=信眾管理
  • rlm define-glossary 奉獻 --description="..."
  • rlm define-story payment-flow --context=payments
  • rlm validate

產出 RLM 變更 PR。

RLM PR review PR

RLM 改動需要審查 (但不像 bootstrap 那樣全人工 — 通常只需要快速 sanity check)：

  • 新 context 邊界是否合理
  • glossary 定義是否與既有詞彙衝突
  • 新 story 是否對齊 #N 的需求

merged 後 → 回到 arch-shape。

arch-shape 重啟，用最新 RLM 分解 arch-shape

workflow 偵測到 RLM PR merged + #N 仍 open + 標 arch-state:awaiting-rlm-update → 重派 arch-shape。

arch-shape 重新查 RLM，現在已有 payments context → 進入分解。

與主流程的接點

分支 A 直接接主流程 Stage 5；分支 B 在 RLM 擴充完成後接 Stage 5。

從 Stage 5 之後 (arch-shape 產出子 issues → executor 並行 → cascade unlock → arch-shape 收尾驗收) 完全相同。

🔁 多次循環　大型專案中 V1 是系統的常態。Stage 1-3 (路由 + 評估) 通常很快；只有分支 B (需要擴充 RLM) 會明顯延長處理時間。

失敗模式

狀況後果處理
arch-shape 誤判，認為現有 RLM 充足，但其實需要擴充 子 issue 的 spec 引用了不存在的 context / glossary slug executor 在 Stage 7 會發現 → 觸發 V6 (RLM drift)
arch-shape 過度擴充 RLM RLM 出現「等以後會用到」的空 context review 時應拒絕投機性 entity；強制要求 story 對應到 #N 的需求
新需求需要 ADR (V4) arch-shape 應該識別並走 V4，而非直接擴 RLM 判準：是否有「兩個合理選項需要選擇」？有則走 V4

為何這樣設計

四角紀律　arch-shape 自己不寫 RLM (沒有寫權)，遇到 RLM 不夠用時必須發信號讓 rlm-modeler 處理。這保證所有 RLM 改動都經過同一個 writer 和 rlm validate，避免概念漂移。

這裡的關鍵設計是「分解前先校準模型」：如果 arch-shape 在 RLM 不足的情況下硬切子任務，產出的 spec 會引用模糊的概念，下游 executor 只能猜，最後實作出來的東西未必對齊原意。

強制走分支 B 看起來慢，但它把「model 變化」和「code 變化」分開審查，每個 PR 都聚焦單一意圖：

  • RLM PR：審查領域邊界 / 概念定義是否合理
  • code PR：審查實作是否對齊 spec
